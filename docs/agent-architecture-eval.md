# Loom Agent 架构评价报告

> 编制日期：2026-07-22
> 评价人：AI 架构师（深度分析）
> 范围：loom-novel v0.4.1，基于源代码（agents.py / journey.py / partner.py / backends.py / gates.py / 15 份 ADR）及主流 agent 架构方案（2025–2026）

## 1. Loom 架构全景简述

### 1.1 核心理念

Loom 是一个**本地优先、零服务器、单机桌面**的长篇小说 AI 写作辅助工具。核心价值在于**越写越像你**——通过写作指纹（只从人类改稿 diff 学习）实现对个人文风的收敛，而不是生成"中性的 AI 大白话"。

### 1.2 当前 Agent 架构

```
┌─────────────────────────────────────────────────────────┐
│                     Loom Agent 架构                       │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  【5 道流水线工序 + 1 个领航员伙伴】                     │
│                                                          │
│  ① 设定师 (Loremaster) —— 立规矩，压设定锚点 ≤350 字     │
│  ② 大纲师 (Outliner)      —— 搭骨架，产出可编辑细纲      │
│  ③ 写手 (Writer)          —— 落字，按写作指纹 + 硬设定   │
│  ④ 编辑 (Editor)          —— 挑硬伤，产出改稿 + 留痕     │
│  ⑤ 润色师 (Polisher)      —— 去 AI 味，产出终稿          │
│                                                          │
│  ⑥ 领航员 (Navigator/Partner)                            │
│     └─ 书房伙伴对话 agent，手搓对话循环                   │
│     └─ 三工具：读文件 / 看地基 / 提设定（候选卡）          │
│                                                          │
│  【质量关卡（Gate）】                                     │
│  编辑后：质检 Gate          ─ 人物 OOC / 设定漂移 / 断钩子 │

│                                                          │
│  【确定性机制】                                           │
│  - 硬设定直送（_hardfacts_for）：原文切片旁路，绕开复述漂移 │
│  - 写盘安全闸（guard.py）：空/残废输出不覆盖用户数据       │
│  - 断点续跑（ledger/ledger.py）：签名比对，跳过未变工序    │
│  - 本地检测器（aitell + fatigue）：句内 AI 翻转 + 跨章复用 │
│                                                          │
│  【写作指纹系统】                                         │
│  - 每本书独立一份，只从 human delta（改稿 diff）学习        │
│  - 播种（seed）从真实样本或欣赏作者的范文借力起点          │

│                                                          │
│  【外置大脑】                                             │
│  每本书独有的状态：世界观 / 人物卡 / 卡章纲 / 写作指纹 /  │
│  立项卡 / 违禁词 / 文风参考 / 状态账本                    │
│  成长机制：learn 后追加 [AI回顾] 和 [AI补充]，不覆盖人写   │
│                                                          │
│  【后端路由表】                                           │
│  九选一：DeepSeek / GLM / Kimi / Qwen / 豆包 / 硅基流动   │
│         / Claude Code / Codex / OpenAI 兼容               │
│  模型可下拉可手填 + 实时拉取可用列表                      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 1.3 关键架构特点

| 维度 | Loom 做法 | 说明 |
|------|-----------|------|
| **编排** | 硬编码工序表（STEPS 元组）+ for 循环 | 拒绝 LangGraph/LangChain，ADR 0013 明确否决 |
| **状态** | 文件即真相（md as source of truth） | 无数据库、无 checkpoint blob；游标可丢弃 |
| **通信** | 顺序管道 + 累积工作区 | 每棒读截至目前全部产物（非纯链式） |
| **对话** | 手搓文本协议工具循环 | 非 MCP/function calling 标准；`用:工具名` + `键:值` |
| **后端** | `Backend` Protocol + 子进程/shell | 极简单轮 `complete(system, user)`，无 streaming tools |
| **可观测** | 事件回调 + 审稿留痕 | 无 OpenTelemetry / trace / span |
| **记忆** | 外置大脑 md + 续跑账本 | 无向量库、无 embedding、无 RAG |

## 3. 业界方案对比矩阵

### 3.1 编排层

| 维度 | Loom (手搓流水线) | LangGraph | OpenAI Agents SDK | CrewAI | AutoGen |
|------|-------------------|-----------|------------------|--------|---------|
| **图模型** | 线性 5 工序 + 2 gate 子循环 | StateGraph（有向图，含环/条件边） | 线性 agent 链 + handoffs | 层级/顺序/协商 | 会话式多 agent |
| **状态持久化** | 文件 md（人可读写）+ JSON ledger | SqliteSaver / PostgresSaver（checkpoint blob） | 内置 RunStore | 内存 + 可选文件 | 内存 + 可选 ZMQ |
| **中断/人介入** | 候选卡 + 拍板确认（两轮间） | `interrupt()`（运行中检查点） | 无内置 interrupt | 人工审核 step | 用户代理节点 |
| **循环能力** | for 循环 + while 工具轮 | 图本身支持环 | 支持循环 agent | 支持迭代 | 内置对话循环 |
| **条件分支** | 无（纯线性） | 条件边（如路由到不同 agent） | 手写 if/else | 条件路由 | 自动对话路由 |

**结论**: Loom 的线性流水线在**小说写作**这个特定领域是合理的——创作流程天然是顺序的（设定→大纲→写→审→润色）。引入状态图（有环/条件分支）对当前场景不带来直接收益，反而会增加 checkpoint 双真相和依赖税的复杂度。

### 3.2 工具调用体系

| 维度 | Loom | MCP v1 | OpenAI Function Calling | Claude Tools |
|------|------|--------|------------------------|--------------|
| **协议** | 自定义文本协议 `用:工具名` | JSON-RPC（标准化） | JSON Schema（自动解析） | JSON Schema + 流式 |
| **注册表** | `partner_tools.REGISTRY` dict | MCP Server 声明 | openai 端自动推断 | 上报 tools 列表 |
| **执行** | 手搓分派（local） | HTTP transport（远程） | SDK 自动分派 | Model 侧 tool use |
| **流式支持** | 无（单块返回） | 有 | 有 | 有 |
| **CLI 后端兼容** | `agent_mode` 护栏开关 + 纯文本协议 | 需 HTTP 端点 | 不兼容 CLI | 不兼容 CLI |
| **工具链** | 单轮 complete + 回喂 ≤6 次 | 长期连接 | 内置自动循环 | 内置自动循环 |

**结论**: Loom 的工具调用是**极简自研**——完全为 CLI 后端（Claude Code/Codex）的纯文本通道量身定制。这种方案的好处是无依赖、零版本漂移；代价是无法利用标准化的工具发现、类型校验和远程服务编排。**MCP 兼容是当前最大的缺失能力之一**——如果能标准化接入 MCP 工具，可以解锁外部数据源查资料、参考作品分析、分卷/设定自动建模等能力。

### 3.3 记忆/状态管理

| 维度 | Loom | RAG (主流) | 向量记忆 | 带外 checkpoint |
|------|------|-----------|----------|----------------|
| **核心存储** | md 文件 + JSON ledger | 向量数据库 + 文档 | 向量 embedding | SQLite/Postgres blob |
| **检索方式** | grep + hardfacts 关键词切片 + 标题解析 | 语义相似度 top-k | KNN/ANN 搜索 | 精确键值查询 |
| **跨章上下文** | 上一章正文(尾 1500 字) + 卡章纲 + hardfacts | 自动检索相关片段 | 无上限上下文窗口 | checkpointer 快照 |
| **人可编辑** | ✅ 全部是人可读写 md | ❌ embedding 不可编辑 | ❌ 黑盒向量 | ❌ blob |
| **随故事增长** | 追加 [AI补充] + [AI回顾] write-once | 自动索引新文档 | 增量插入 | 覆盖写入 |
| **依赖** | **零额外依赖** | 向量库 + embedding 模型 | embedding + 索引库 | DB driver |

**结论**: Loom 的"文件即真相"方案在小型长篇小说（≤数百章）场景下极其优雅。但当正文规模增长时，纯 grep + 关键词切片的检索方式会退化。**当前刻意不做向量 RAG 是正确的决定**（ADR 0015 明确红线），但**缺少一个可选的全文搜索（grep/fuzzy find）工具**，是当前能力矩阵的明显空白。

### 3.4 Agent-to-Agent 通信

| 维度 | Loom | Google A2A | OpenAI Agents handoff | AutoGen |
|------|------|-----------|---------------------|---------|
| **通信方式** | 文件累积（工作区） | HTTP JSON-RPC | Agent 间消息传递 | 消息总线 |
| **API 标准化** | 无 | 标准化 A2A（Card + Task） | SDK 内部协议 | ZMQ/grpc |
| **跨进程** | ❌ 同一进程 | ✅ 可跨网络 | ❌ 同进程 | ✅ ZMQ |
| **协商** | 无 | 能力卡（Agent Card） | Handoff 声明 | 群聊协商 |
| **适用性** | 写作流水线（足够） | 企业服务编排 | 动态子 agent | 研究/模拟 |

**结论**: Loom 的 Agent 间通信方式（工作区累积 + 工程化编排）在**单一流水线场景**下是够用的。五个 Agent 之间的"通信"本质上是同一个 prompt 模板中 `reads` 声明的文件引用关系，不需要 runtime 消息传递。但**如果未来要支持"查资料 agent"、"分卷 arc agent"、"角色一致性 agent"等多角色协作**，手搓的文件通道将难以应对——这时 A2A 类协议会显现价值。**目前不需要，中期可以考虑。**

### 3.5 可观测性/调试

| 维度 | Loom | OpenTelemetry | LangSmith/LangFuse | 自研 trace |
|------|------|---------------|-------------------|-----------|
| **追踪** | 事件回调（dict，无 span tree） | Span/Trace 树 + 自动传播 | 完整 trace + token 计数 | 需自建 |
| **审计** | 审稿留痕（文件追加） | 无专门审计 | 对话历史 + 人工反馈 | 需自建 |
| **token 计量** | ❌ 不做 | 无 | ✅ 内置 | 需自建 |
| **延迟分析** | ❌ 不做 | ✅ 内置 | ✅ 内置 | 需自建 |
| **错误归因** | human-readable 错误码 | 异常 + span | 自动标注 | 需自建 |
| **离线可用** | ✅ 全本地 | ❌ 需 collector | ❌ SaaS | ✅ |

**结论**: Loom 的"事件回调 + 审稿留痕"方案是极简但功能性的——它提供了用户级别的审计（改了什么、留了什么），但**不提供开发级别的调试洞察**（哪个环节最慢、每次调用花了多少 token、错误分布的根因）。这是**高质量 agent 系统最明显的缺失能力**，对基于 API-KEY 的自费用户尤其重要——没有 token 计量，用户无法感知"哪个 agent 吃了我最多的 API 额度"。

### 3.6 事件驱动/观察者模式

| 维度 | Loom | Event-Driven Agent |
|------|------|-------------------|
| **事件机制** | 简单回调（dict 参数） | Pub/Sub / Event Bus / Webhook |
| **订阅能力** | 无（调用方必须实现回调） | 可插拔监听器 |
| **事件持久化** | 实时 emit + 部分文件落盘 | 事件溯源（Event Sourcing） |
| **消费者** | 前端 UI 渲染（唯一） | 多消费者：审计/监控/触发/重试 |

**结论**: Loom 的事件系统是 UI 驱动的——回调只服务于前端渐进渲染。缺乏独立的事件总线意味着无法在不修改核心代码的情况下加入新的消费者（如日志、成本统计、自定义 post-processing hooks）。

## 4. 缺失能力清单（按优先级）

### 🔴 HIGH（必须补充或认真考虑）

| # | 缺失能力 | 当前状态 | 影响 |
|---|---------|---------|------|
| G1 | **用户可自定义审查 Gate** | Gate 的 rubric 硬编码在 `gates.py` | 用户反馈 F1 明确要求审查节奏/情绪/钩子，目前无法自定义。F2 用户说"要自己会调教"也指向这方面 |
| G2 | **Token 计量与成本可观测** | 完全不做 | BYOK 用户无法追踪"哪个 agent 吃了我多少钱"。用户感知成本不可见会降低使用意愿 |
| G3 | **每步调用延迟/Trace** | 事件回调不携带时序/费用元数据 | 用户无法定位"哪一步最慢/最费钱"，调试体验差 |
| G4 | **Gate 结果结构化回馈** | Gate 硬伤只在审稿留痕中追加文本 | 用户反馈的"检查节奏情绪钩子"需要结构化可视化，不只是文本附录 |

### 🟡 MEDIUM（可放在中期路线图）

| # | 缺失能力 | 当前状态 | 影响 |
|---|---------|---------|------|
| G5 | **全文检索（grep / fuzzy find）工具** | 只有关键词切片 + 上一章尾部 | 写手生成时无法搜索全书已有的线索/设定，跨章一致性靠 hardfacts + 卡章纲，数百章后可能不够 |
| G6 | **标准工具协议（MCP 兼容）** | 自定义 `用:` 文本协议 | 无法接入外部数据源（查梗、查资料）。CLI 后端无法享用标准工具调用 |
| G7 | **Gate 扩展接口（plugin hook）** | gate 循环是私有函数 | 第三方或高级用户无法写自己的质量关卡 |
| G8 | **Agent 互访/委托（dynamic delegation）** | 固定 5 道工序，不支持运行时选择/替换 | 无法在写作过程中按需召唤"查资料 agent"、"设定审计 agent"、"匹配风格 agent" |
| G9 | **分卷/arc 层级支持** | CONTEXT 明确拒绝"卷"概念 | 用户写长篇时缺乏结构性 arc 管理。卡章纲的 H2 `## 第N卷` 可看但不作为运行时的一级 |
| G10 | **增量 / 局部重写** | 当前只能整章重写（全 5 工序） | 用户改了设定后只想刷新某几章的部分段落，必须整章重跑 |

### 🟢 LOW（长期考虑，非紧迫）

| # | 缺失能力 | 当前状态 | 影响 |
|---|---------|---------|------|
| G11 | **向量库/RAG 检索** | ADR 红线：不建 | 当前 md + grep 足够，但当全书超 500 章后可考虑可丢弃索引 |
| G12 | **A2A 协议支持** | 无跨进程 agent | 目前单体应用不需要 |
| G13 | **事件溯源/审计日志** | 现有审稿留痕足够 | 强审计需求的环境（出版行业）可以考虑 |
| G14 | **A/B 比较生成** | 无 | 用户想同时跑两个不同的指纹/设定配置对比效果时有用 |
| G15 | **CLI 后端 streaming** | claude/codex 后端非流式 | 不影响正确性，但前端体验弱于流式 |

    ├── → G1 (HIGH): 自定义能力 → 用户可自己编写审查正则/提示词

         当前：固定 5 道工序不受 runtime 影响
         愿景：用户在对话中要求"帮我查一下第5章埋的伏笔"，领航员调度一个"查伏笔 agent"

F3: "很多收费去ai软件不挺多的吗"
    ├── → 竞争差异化（非技术架构问题）
    └── → 产品定位：Loom 的差异化在于"本地优先 + 写作指纹收敛到你" + "越写越像你"
         架构短板：去 AI 味 Gate 的 "确定性预筛" 目前只有 aitell + fatigue 两个本地检测器

### 5.2 核心矛盾剖析

从反馈 F2 可以读出更深层的矛盾：

> **用户感知：Loom 的 Agent 功能可被通用 API + agent 替代**
>
> 这表明 Loom 的"5 道工序编排"在高级用户眼中不是不可替代的护城河。用户如果熟悉 prompt 工程，自己用 API 搭一个类似流水线也不难。
>
> **真正的护城河在哪里？**
> - 写作指纹（human delta 学习）——这是通用 API 搭不出来的
> - 硬设定直送 + 状态账本——确定性保障，通用 API 不会帮你做
> - 本地优先 + 零服务器——隐私保障
> - 全流程的"不阻断"哲学——让作者始终握着终审权
>
> **架构启示**：Loom 应该把 agent 编排层做得更加**可观察、可自定义、可扩展**，而不是进一步硬化。越是让用户感到"Loom 的 agent 我可以改、可以调、可以看到它在干什么"，防御力越强。

### 6.4 不建议做的（已否决或不应做）

| # | 不建议项 | 原因 |
|---|---------|------|
| N1 | **引入 LangGraph / LangChain** | ADR 0013 已清算：依赖税（+26MB）、checkpoint 双真相、interrupt 无执行栈可挂、单人维护不划算。**维持否决** |
| N2 | **引入向量 RAG 作为真相** | ADR 0015 红线：embedding 是文件内容的派生缓存，作者改 md 即馊，且无法用 sig 哈希兜底。**维持否决** |
| N3 | **中央服务器 / 账号系统** | 与"本地优先、零遥测"的核心定位冲突。Loom 无服务器是卖点 |
| N4 | **逐 agent 模型矩阵（inkos 风格）** | CONTEXT 明确否决：粗粒度一刀切即可，"省钱省到像你头上"是错误方向 |
| N5 | **分数驱动的 Gate 硬阻断** | ADR 0006 明确否决：因果错位、不可自检。**维持"硬伤清单空不空"判据** |
| N6 | **用 NLU/embedding 替代 hardfacts 关键词切片** | ADR 0010 红线：确定性零成本方案足够，LLM 切片有失真风险 |
| N7 | **写作指纹绑检测器分数** | ADR 0002 撤回决定：过检测是副作用，不是指纹 KPI |

## 7. 总结

### 7.1 架构总体评价

Loom 的 agent 架构是**领域专用、高度自洽的手搓系统**。它的每一个设计选择（不引框架、文件即真相、顺序流水线、硬编码 Gate、自定义文本协议）在**长篇小说写作**这个具体场景下都有充分的、经过书面论证的理由（15 份 ADR）。**这不是通用 agent 框架**，也不想成为通用 agent 框架——它是为"让 AI 写出像你的小说"这个单一目标而做的工程优化。

### 7.2 最大优势

- **文件即真相**：人可读可改，不会有"用户在 UI 里改了一个值，但 checkpoint 闭包还是旧值"的双真相地狱
- **写作指纹系统**：只学 human delta，是通用 API 很难复制的差异化壁垒
- **确定性保障链**（hardfacts 直送 + guard 写盘安全闸 + 续跑签名 + 原子写），在"AI 输出不可靠"的现实下提供了工业级稳定性
- **全流程"不阻断"哲学**：用户始终拿得到一份完整的输出 + 留痕清单，符合创作场景的人机协作

### 7.3 核心短板

- **可扩展性不足**：Gate rubric 硬编码、工序不可插拔、工具协议不标准——高级用户或第三方无法轻易定制
- **可观测性缺失**：无 token 计量、无调用 trace、无延迟分析——BYOK 用户看不见花费，调试靠猜
- **确定性检索受限**：关键词切片 + 上一章尾部 1500 字，超过数百章后会遇到瓶颈
- **对话 agent 工具集偏小**：只有 3 个工具（读文件/看地基/提设定），没有搜索/计算/联网能力

### 7.4 针对用户反馈的最终建议

> **F1 "加个ai审核提示词"** → 短期可做（S1+S2），是最紧迫的架构优化项
>
> **F2 "代理功能可被通用agent替代"** → Loom 需把护城河建立在"写作指纹 + 确定性保障 + 本地优先"而不是"5 道工序编排"上；中期可通过 M1 + M3 + M6 让用户自定义 agent 行为
>
> **F3 "收费去ai软件很多"** → 去 AI 味 Gate 可通过 S4 扩充本地检测器+ S1 + M1 自定义 rubric 实现差异化
>
> **F4 "联系我修bug"** → 降低运营负担的方法是提高可观测性（S3 + trace），让用户自己看的见问题
