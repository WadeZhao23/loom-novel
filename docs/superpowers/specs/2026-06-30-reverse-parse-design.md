# 小说反向解析设计规格

## 背景

Loom 目前只能从空项目开始创作，不能把已有 TXT 小说转换为可继续维护的 Loom 项目。本功能提供持久化导入任务：先在本地拆分章节并由用户确认边界，再按选定范围调用 AI 提取世界观、力量体系、人物卡和卡章纲，最后由用户修订结果并创建、注册项目。

本规格建立在现有全局 API Key、项目注册表和章节规划能力之上，并保持已有 API 与创作流水线不变。

## 已确认的产品决策

- 导入任务支持断点恢复，关闭或刷新页面不丢失进度。
- 上传后先预览、编辑并确认章节边界，再调用 AI。
- 章节默认全选，用户可只解析连续范围或手工选择章节。
- 创建项目时可选择“写入分章原稿”或“仅写入提取结果”。
- 源 TXT、中间结果和最终结果一直保留，直到用户主动删除任务。
- 使用文件化任务目录，不引入数据库或独立 worker 进程。

## 目标

1. 支持上传不超过 50 MiB 的中文 TXT 小说，并可靠处理 UTF-8 BOM、UTF-8 和 GB18030。
2. 本地拆章后允许改标题、拆分、合并、排序和选择解析范围。
3. 对长篇小说采用分块提取与归并，不把全书一次发送给模型。
4. 每个阶段和分块完成后立即落盘，失败后只重跑未完成或已失效工作。
5. 用户可编辑所有解析结果，并从同一任务重复创建项目。
6. 项目完全写入并通过结构验证后才加入全局项目注册表。

## 非目标

- 不支持 EPUB、DOCX、PDF 或压缩包。
- 不提供云端任务队列、多用户权限或跨设备同步。
- 不自动覆盖已存在的项目目录。
- 不修改现有设定师、大纲师、写手、编辑和润色师流水线。
- 不在本期实现任意树状章节层级；章节保持有序平面列表。

## 总体架构

### `loom/import_jobs.py`

负责持久任务仓库，不包含 AI 或项目创建逻辑：

- 创建、读取、列表和删除任务。
- 校验任务 ID，所有路径限制在导入根目录中。
- 原子写入 JSON 和文本文件。
- 为每个任务提供进程内互斥锁。
- 启动时把遗留 `running` 状态归一为 `interrupted`。

默认根目录为 `~/.loom/imports`。测试可通过显式参数或环境隔离替换该目录。

### `loom/reverse_parse.py`

负责文本和 AI 解析：

- 解码与换行标准化。
- 章节标题识别和无可靠标题时的候选分块。
- 章节编辑操作及内容摘要计算。
- 选中章节的分块、局部提取、阶段归并和检查点恢复。
- 通过 `progress(event)` 回调报告结构化进度。

### `loom/import_project.py`

负责把已确认结果物化为 Loom 项目：

- 调用 `scaffold.init()` 创建标准骨架。
- 写入世界观、力量体系、人物卡和卡章纲。
- 按创建选项决定是否写入分章原稿。
- 调用服务层结构检查，成功后才调用项目注册表。
- 创建失败时不注册，并保留导入任务供修正后重试。

### `loom/server.py`

仅承担 HTTP 参数校验、状态码、NDJSON 桥接和服务调用。静态资源挂载继续保持在所有 API 路由之后。

### `loom/webui/`

沿用现有无框架 JavaScript 和覆盖层模式，在欢迎页增加导入入口与历史任务列表。工作流由四个稳定步骤组成：上传、章节确认、解析、结果与创建。

## 持久化格式

每个任务位于 `~/.loom/imports/<task_id>/`：

```text
<task_id>/
  source/
    original.txt       # 上传的原始字节
    normalized.txt     # 统一为 UTF-8 与 LF 的文本
  task.json            # 状态、阶段、选择、错误与时间戳
  chapters.json        # 可编辑章节列表
  runtime/
    loom.toml          # 从模板复制，仅用于加载全局 AI 配置
  checkpoints/
    worldview.json
    system.json
    characters.json
    outlines.json
  results/
    worldview.md
    system.md
    characters.md
    outlines.md
```

任务 ID 使用服务端生成的 UUID，不接受用户提供的目录名。所有 JSON 先写同目录临时文件，再通过 `os.replace()` 原子替换。

`task.json` 至少包含：

```json
{
  "id": "uuid",
  "original_filename": "book.txt",
  "encoding": "gb18030",
  "status": "reviewing",
  "phase": null,
  "chapter_revision": 1,
  "result_revision": null,
  "selected_chapter_ids": [],
  "progress": {"completed": 0, "total": 0},
  "error": null,
  "created_at": "ISO-8601",
  "updated_at": "ISO-8601"
}
```

任务状态为：

- `reviewing`：已拆章，等待用户确认。
- `ready`：章节和范围已确认，可开始解析。
- `running`：后台解析中。
- `interrupted`：进程中断，可从检查点恢复。
- `failed`：阶段失败，错误已持久化，可重试。
- `completed`：所有结果可预览和编辑。
- `created`：至少成功创建过一个项目；任务仍可再次创建项目。

`chapters.json` 中每章含稳定 UUID、顺序、标题、正文、字符数和 `selected`。合并、拆分和正文修改都会增加 `chapter_revision`。若结果对应的 `result_revision` 不再等于当前章节修订号，结果标记为过期，禁止直接创建项目。

## 上传与章节拆分

上传接口使用 multipart，因此增加 `python-multipart` 运行依赖。服务端流式读取并执行 50 MiB 上限，不把无限大小文件一次读入内存。

解码顺序为 UTF-8 BOM、严格 UTF-8、严格 GB18030；全部失败时返回 400。解码后统一换行为 LF，拒绝空文本和包含大量 NUL 字节的二进制内容。

章节标题检测覆盖常见形式，例如“第十二章 标题”“第12回”“卷一”“楔子”“序章”“后记”。只有候选标题数量和正文覆盖率达到内部阈值时才视为可靠拆分。否则按段落和字符预算生成候选章节，并在任务中设置 `split_confidence: "low"`，要求用户检查后才能解析。

章节编辑支持：

- 修改标题或正文。
- 在光标位置拆分为两章。
- 将一章与上一章或下一章合并。
- 上移、下移和重新编号。
- 全选、连续范围选择和逐章勾选。

每次保存都校验至少保留一章、标题非空、顺序唯一、选中集合非空。

## AI 解析流水线

导入任务的 `runtime/loom.toml` 从内置模板复制，使现有 `load_config()` 和 `get_backend()` 能直接复用。任务目录没有项目级 `.env`，因此使用全局 API Key；后端和模型沿用模板默认值。

选中章节按字符预算组合成输入块。每个块保存内容摘要，检查点同时记录阶段、块编号、输入摘要和输出。恢复时只有输入摘要一致的已完成块可复用。

流水线顺序：

1. `worldview`：逐块提取地域、组织、规则、历史和核心冲突，再全局归并去重。
2. `system`：逐块提取修炼或力量层级、资源、限制和晋升规则，再全局归并。
3. `characters`：逐块提取人物身份、关系、目标、能力和状态，再按人物归并。
4. `outlines`：按章节生成卡章纲摘要，保留章节 UUID 与标题对应关系。

世界观、体系和人物阶段采用“局部提取 → 分批归并 → 最终归并”；章纲按章独立生成。每次 `Backend.complete()` 都有明确字符上限，避免整本上下文溢出。

进度事件至少包含：

```json
{"type":"phase","phase":"worldview","message":"正在提取世界观"}
{"type":"progress","phase":"worldview","completed":2,"total":8}
{"type":"phase_done","phase":"worldview"}
{"type":"complete","task_id":"uuid"}
{"type":"error","phase":"characters","message":"..."}
```

后台 worker 与 HTTP 响应解耦。浏览器断开时 worker 继续运行并写入任务状态；刷新后通过任务详情恢复。服务进程退出导致的 `running` 状态在下次读取时转为 `interrupted`。

同一任务同一时间只允许一个解析 worker。重复启动正在运行的任务返回 409 和当前任务快照，不产生重复 AI 调用。

## HTTP API

### 任务管理

- `POST /api/imports`：multipart 上传，返回 201 与任务详情。
- `GET /api/imports`：按更新时间倒序列出任务摘要。
- `GET /api/imports/{task_id}`：返回任务、章节摘要和结果可用性。
- `DELETE /api/imports/{task_id}`：仅非运行任务可删除；成功返回 204。

### 章节与解析

- `PUT /api/imports/{task_id}/chapters`：保存完整章节列表和选中集合，返回新修订号。
- `POST /api/imports/{task_id}/parse`：开始或恢复解析，返回 `application/x-ndjson`。
- `PUT /api/imports/{task_id}/results`：保存用户编辑后的四类结果。

解析流启动前的校验错误使用 400、404 或 409。流启动后的异常写入 `type=error` 帧并持久化到任务状态。

### 创建项目

- `POST /api/imports/{task_id}/create-project`

请求体：

```json
{
  "name": "项目名",
  "parent_dir": "E:/小说",
  "genre": "玄幻",
  "include_source_chapters": true
}
```

结果写入：

- `外置大脑/世界观.md`
- `外置大脑/修炼体系.md`
- `外置大脑/人物卡.md`
- `外置大脑/卡章纲.md`
- 选择写入原稿时：`正文/.原稿/<序号>-<安全标题>.md`

新增模板 `loom/templates/外置大脑/修炼体系.md`。文件名中的非法字符被替换，序号固定宽度，正文内容保留原章节标题。

若目标目录存在则返回 409，不覆盖。写入和验证全部成功后注册项目并返回现有 `_state()` 结构。失败项目目录保留供用户检查，但不注册；响应明确返回失败路径。

## 前端交互

欢迎页在“新建项目”和“打开项目”附近提供“导入小说”命令，并显示历史任务列表。任务项展示文件名、章节数、状态、更新时间和继续/删除操作。

导入覆盖层包含四步：

1. 上传：拖放区和文件选择器，只接受 `.txt`。
2. 章节：紧凑表格展示选择框、序号、标题、字符数和拆分/合并/排序操作；提供全选与范围输入。
3. 解析：固定阶段列表、总进度和当前错误；失败时提供恢复按钮。
4. 结果：世界观、力量体系、人物卡、卡章纲标签页及可编辑文本区；创建区使用目录、题材和正文模式控件。

页面只展示操作所需文本，不加入营销式说明。所有按钮在请求期间有稳定尺寸和禁用状态。刷新后按任务状态恢复正确步骤；创建成功后调用现有 `enterProject(state)`。

## 并发、错误与安全

- 任务级锁覆盖章节保存、解析启动、结果保存、创建和删除的状态转换。
- 运行中的任务不能编辑章节、创建项目或删除。
- 结果过期、结果缺失或解析未完成时不能创建项目。
- 所有用户可控路径经过现有安全路径与项目目录校验。
- 上传文件名仅作为显示元数据，不参与磁盘路径拼接。
- API 延续现有本机 Origin/Referer 写请求保护。
- AI 错误不删除检查点；重试从最后有效检查点继续。
- 删除是唯一清除任务源文件和中间结果的操作，前端必须二次确认。

## 测试策略

### 单元测试

- UTF-8 BOM、UTF-8、GB18030、空文件和二进制拒绝。
- 常见章节标题、低置信度回退、标题误判。
- 章节改名、拆分、合并、排序和范围选择。
- 修订号、结果失效、原子写入和任务 ID 路径保护。
- 分块预算、检查点摘要匹配、失败恢复和阶段归并。

### API 测试

- multipart 上传大小与类型限制。
- 任务 CRUD、章节保存和结果编辑。
- NDJSON 事件顺序、流中错误和断点恢复。
- 重复启动、运行时删除、运行时编辑等 409 冲突。
- 服务重启后 `running` 转为 `interrupted`。

### 项目创建测试

- 两种正文模式对应的文件布局。
- 四类解析结果写入正确路径。
- 已存在目录不覆盖。
- 写入或验证失败时不注册。
- 成功后注册并返回可打开的项目状态。

### 前端与最终验证

- 手工和浏览器自动化覆盖上传、章节编辑、范围选择、恢复、结果编辑、两种创建模式和删除确认。
- 检查桌面与窄视口无文本溢出、控件重排或遮挡。
- 完整运行 `pytest`、Python 编译、`node --check loom/webui/app.js`、浏览器端到端流程和 `git diff --check`。

## 兼容性

所有新增 API 位于 `/api/imports` 下，不修改现有响应格式。新模板只影响新建项目；已有项目无需迁移。导入任务位于用户目录，不进入 Git 仓库，也不改变项目注册表格式。
