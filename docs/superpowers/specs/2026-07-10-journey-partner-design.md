# 全程写作伙伴(访谈式创作旅程)——设计

2026-07-10 · 已与作者逐节确认。多 agent 管理创作旅程状态;架构选型经四视角分析(产品形态/LangGraph 钢人/工程经济/旅程图复杂度)裁决。

## 0. 一句话

给 Loom 加一层**薄的旅程编排层**:「领航员」agent 用结构化问题卡逐步访谈作者,答案经确认后直接落进外置大脑 md——作者在回答问题的过程中完成起书与逐章创作决策;既有流水线一个字节不动。

## 1. 需求(已拍板)

- **范围**:全程写作伙伴——起书访谈(立项/世界观/人物/卡章纲/voice 逐步问出来)→ 每章开写前 2-3 个关键剧情决策 → 写完引导手改 + learn。
- **交互面**:桌面端常驻「伙伴面板」;CLI 暂不做。
- **问题形态**:结构化卡片(一题 + 2-4 个 AI 预填选项 + 「自己写」输入框),HTTP 一问一答。

## 2. 架构裁决:手搓旅程状态机,照抄 LangGraph 的协议形状

用户最初提议 LangGraph。四视角分析(全文见会话记录,要点如下)裁决为**不引入框架**:

- **产品形态推出三条红线**:①不得有第二份状态真相(访谈产出归宿只能是人可读可改的 md);②必须容忍用户绕过旅程直接改文件(细纲 WYSIWYG 判例);③不能依赖内存执行上下文活过两次问答之间(进程随时死)。LangGraph 的 checkpointer(msgpack 二进制 blob)三条全撞:人改了 md 它即刻过期,防漂移唯一办法是每节点从 md 重新 hydrate——做到那一步框架已沦为摆设。
- **interrupt() 无用武之地**:它解决「挂起运行中的执行栈等人」;Loom 的 HTTP 一问一答之间没有任何挂着的计算,要持久化的只是一个 3-4 字段游标。且 interrupt 节点 resume 重放会造成 LLM 重复计费(BYOK 用户直接看账单)。
- **工程经济(实测 langgraph 1.2.9)**:净新增 18 包/26MB(现有运行时直接依赖仅 7 个),Loom.app 74MB → 估 ~95-100MB;langsmith 遥测客户端被无条件拉入(零联网产品的审计面);pydantic<3 上限锁 fastapi 升级;claude/codex CLI 通道协议层无 tool-call,框架预制件对一半后端失效;年 2-3 波破坏性变更 × 5 包 × 双平台打包税由一人扛。
- **旅程图复杂度实测**:12 个静态宏节点、~15 条边、0 并行、0 运行时动态路由、2 个 while 可表达的环——远低于图框架回本线。最吓人的转移「写到第 10 章回头改世界观」在文件真相架构里是 **0 条边**:改 md → 设定师 reads 签名失配 → 下次织章自动全量重跑,失效传播免费。
- **照抄的协议形状**(LangGraph 值得偷的四个思想):问题卡=纯数据 payload(不是代码分支);答案经单点注入(`POST /api/journey/answer`);游标持久化;「改答案」=改文件即 fork。

## 3. 架构总览

```
伙伴面板(webui 卡片 UI)
    │  HTTP 一问一答
    ▼
journey.py ── 旅程状态机(STAGES 冻结表 + 完成谓词 + 薄游标)
    │  出题时调              │  答案落盘时调
    ▼                        ▼
领航员 agent            guard + 原子写 → 外置大脑 md(单一真相)
(第六个角色)                │
                             ▼
                  既有引擎原样不动:run_pipeline / ledger / gates / learn
```

三层状态各管各的:**领航员管旅程**(问什么、何时问)→ **五工序管织章**(章内状态归 ledger)→ **复审员管质量**(gate 回炉环)。旅程层把「织章」当不透明节点,绝不伸进工序内部。

## 4. 旅程状态模型

**原则:出题以文件现状为准,不以问答历史为准。** 领航员出题前看外置大脑 md 的当下内容,不看「上次问到第几题」。由此免费获得:用户外改文件天然支持;游标可丢弃(坏了从文件谓词重推到段粒度,最坏重问一题,答案幂等无害);「回头改答案」=改文件,无需 fork 机械。

**STAGES 冻结表**(照抄 agents.py 的 STEPS/StepSpec 模式,拓扑住代码侧、不下放用户可编辑文件):

| 阶段 | 完成谓词(确定性、文件派生) | 落盘目标 |
|---|---|---|
| 立项 | 立项卡非模板 | `外置大脑/立项卡.md` |
| 世界观 | brain_ready 式判定 | `外置大脑/世界观/*.md` |
| 人物 | 人物卡存在且非模板 | `外置大脑/人物/*.md` |
| 卡章纲 | 有实质章行 | `外置大脑/卡章纲.md` |
| voice(可选) | fingerprint_source ≠ default | 指纹 / `文风参考.md` |
| 逐章循环 | 章前决策=该章 `[决策]` 子块存在;写完=正文存在;手改=edited 徽标;learn=state.learned | 见 §6 |

**薄游标**:挂在既有 `.loom_state.json` 的 `journey` 键下,字段仅:当前阶段、本段已问题数(执行每段题数预算:**默认封顶 4 题**,代码侧常量、不进 loom.toml 加螺丝)、阶段跳过标记(`{"世界观":"skipped"}`——用户说跳过后,谓词判未完成也不再追问)、待答卡缓存(题面+选项+输入签名,防关窗重开重复计费出题)。

**阶段推进的三个触发**,全部确定性:用户点「下一段/跳过」;本段题数预算用尽;领航员出题返回「本段无题可问」。

## 5. 领航员 agent(第六个角色)

一段系统提示词 + `reads:` 清单(`agents/领航员.md`),无会话态,每题一次 `complete`。

- **职责边界**:只出题、只消化答案成落盘文本,**绝不发明设定**——预填选项是给作者拍板的提案,作者选中/改写/自己写才落盘(守「loom 不发明设定、人主 AI 佐」)。
- **出题**:输入=当前阶段目标 + 该阶段 md 现状 + 已答预算;输出=一张卡的 JSON(一题 + 2-4 选项 + 允许自由输入),解析走 parse.py 模式。
- **章前决策卡**:读卡章纲该章行 + 上一章结尾 + 状态账本存活快照,出 2-3 个关键岔路题(视角/转折点/伏笔兑现)。
- **模型路由**:出题/消化是「管 what 的评估类」调用 → 走既有 `cheap_model` 通道(空则主模型);写作与学指纹永远不经它。

## 6. 答案落盘通道

- **起书访谈答案**:作者拍板过的答案=**人写主体**,经 guard + 原子写直接写进对应 md。整形分两档,由 STAGES 表里该卡的 `target` 声明、代码侧定死:**简单目标模板化直写零 LLM**(立项卡格子/卡章纲行);**复杂目标一次消化调用**(答案整形成世界观小节散文块)。**消化只整形作者答案的表述,不得添加作者未确认的设定事实。**
- **与 ADR 0011「立项卡只读不回写」的边界**:该红线禁止的是 loom **自动**写卡(联网拉分区、AI 替填);访谈是**作者逐格口述、工具代为落盘**,主语是人——同一语义下建书框收书名/题材已是先例。世界观/人物卡同理:访谈落盘的内容属人写主体、不是 `[AI补充]` 块,ADR 0007 的「AI 只追加隔离子块」约束管的是 learn 自动蒸馏,不管作者拍板的答案。
- **voice 段**:卡片引导作者贴 2-3 段真实样本(或跳过用中性默认指纹),落盘走**既有 seed 通道**(联网、guard 已罩),不新发明指纹写路径。
- **章前决策答案(已拍板选甲)**:落**卡章纲该章行下的 `[决策]` 缩进子块**——格式类比 `[AI回顾]` 但语义是人写(作者拍板的决策)。大纲师本来就读整份卡章纲,答案自然进入细纲生成;细纲 WYSIWYG 语义零干扰;子块随删章/重编号走 chapters.py 既有搬运。write-once 判重照抄 `[AI回顾]` 模式。
  - _否决的乙案_:答案预生成细纲草稿——细纲存在即触发 WYSIWYG 旁路,大纲师会把半成品决策记录当成品细纲直接用,不做。

## 7. 端点与前端

4 个端点,全走既有 JSON 端点模式;写锁复用 usecases 的 per-root 锁(织章期间答题 409 → 面板转「织章中」视图):

| 端点 | 做什么 | LLM |
|---|---|---|
| `GET /api/journey/state` | 派生谓词 + 游标 + 缓存卡 → 面板一次渲染 | 0 |
| `POST /api/journey/card` | 出下一张卡;缓存卡输入签名(源 md hash)未变直接回缓存 | ≤1(cheap) |
| `POST /api/journey/answer` | 收答案 → 整形 → guard+原子写 → 清缓存、推游标 → 回新 state | 0-1 |
| `POST /api/journey/goto` | 回跳到某段 / 跳过当前段(写 override) | 0 |

**伙伴面板**:webui 常驻面板——六段进度条 + 当前卡片(题面/选项/自己写/跳过)+「下一段」「回头改」。它是现有五步旅程卡(app.js:693,已全派生谓词)的交互化升级,收编不另起炉灶。逐章期给引导位:「第 7 章还没过手改」「第 6 章可以 learn 了」。

## 8. 错误处理

- **出题失败**(断网/超时/烂 JSON):卡片降级为「自由输入 + 跳过」,旅程永不卡死(ADR 0003 失败隔离哲学);错误经 errors.py 目录友好化。
- **落盘**:空/残废答案被 guard 拦下保留旧文件;全部原子写;`[决策]` write-once 判重。
- **游标损坏**:当无游标,文件谓词重推到段粒度,缓存卡作废重出(「坏 JSON 当无 ledger 重跑」同款)。
- **计费保护**:缓存卡 + 输入签名防重复出题;出题/消化走 cheap_model。

## 9. 测试

- journey.py 纯函数状态机:STAGES 谓词/推进触发/丢游标重推,pytest 直接钉(test_statebook.py 粒度)。
- 领航员烂响应/半 JSON/空响应 → 降级路径断言(fixture 化)。
- `[决策]` 子块判重/删章清理/重编号搬运,挂进 chapters 既有测试族。
- `agents/领航员.md` 进 golden 快照;新事件类型进 events.py 注册表 + 契约测试。

## 10. 分期

1. **一期(独立可发版)**:journey.py + 领航员 + 起书访谈五段 + 伙伴面板——新书冷启动从「面对空模板」变「答题起书」。
2. **二期**:章前决策卡(`[决策]` 子块)+ 写后引导(手改/learn 提示位)。
3. **三期(可选)**:goto 回改体验打磨、按已写正文反推追问。

## 11. 刻意不做(Avoid)

- 引入 langgraph/langchain 任何包(裁决见 §2;若他日真出现高分支/并行需求,先怀疑违反了「交互从简」再考虑框架)。
- 旅程状态的第二真相(sqlite/二进制 checkpoint/独立会话库);游标之外一切状态派生自文件。
- 领航员发明设定或未经作者确认落盘;自由聊天式访谈(卡片为主,守交互从简)。
- 访谈答案回流写作指纹(学习信号永远只是改稿 diff,ADR 0002)。
- 把问答历史当真相(出题只看文件现状);把细纲当决策落盘点(乙案已否决)。
- CLI 访谈通道(暂不做,伙伴面板先行)。
