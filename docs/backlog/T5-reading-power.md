## 问题(Loom 现状)

Loom 的追读力手艺只覆盖"大爽点 + 章末钩"两层，缺中间的"章内微兑现"层和"钩子按位置分工"的概念，章衔接的兑现也没进自检。
现状证据：
- /Users/chambers/Desktop/Project/playground/Loom/loom/templates/skills/故事引擎.md:6-9 的"情绪工程"只讲"蓄压→爆发"的大节奏，整章只要求一个完整爽点，没有"章内每章 1-2 个小收获"这一层，过渡章/铺垫章容易写成"读者看完没收获"。
- 故事引擎.md:18 的"章末钩子"把所有钩混在章末讲，没区分"章内钩(保持沉浸) vs 章末钩(驱动点下一章)"的职能；章首(故事引擎.md:17)只讲"接住上一章钩子"，但没在自检里被验证。
- /Users/chambers/Desktop/Project/playground/Loom/loom/templates/skills/评估自检.md:7-14 的是非题里有"爽点/钩子/章首衔接"三条，但：① 没有"这章有没有微兑现/小收获"这一条；② "章首衔接"(评估自检.md:9)只问"有没有接住 + 有没有拖沓回顾"，没问"上一章埋的章末钩，这一章实质接住/兑现了吗"(即承诺兑现)。
后果：单章可能"大爽点合格但通篇没小收获"、过渡章空转、上一章的钩子在这一章被悄悄丢弃而无人发现——这正是追读力流失的三个典型缺口。

## 从 webnovel-writer 借什么 / 丢什么

从 /tmp/webnovel-writer/webnovel-writer/references/reading-power-taxonomy.md 只取三块【定性概念】：
1. 微兑现定义 + 类型(§3.1-3.2, 行 245-263)："章节内给读者的'小收获'，让读者感觉'这章没白看'…比大爽点更轻量、更频繁，维持章内沉浸"，7 类型(信息/关系/能力/资源/认可/情绪/线索兑现)。
2. 微兑现的题材偏好数量 + 过渡章降级(§3.3 行 267-272、§3.4 行 287-299)："每章建议 1-3 个；过渡章优先保留至少 1 个微兑现；软提示问句：读者看完这章会获得什么？是否有'这章有收获'的感觉？"
3. 章内 vs 章末钩分工(§1.4 行 150-156)："章内→悬念钩/情绪钩，保持章内沉浸；章末→危机钩/选择钩/渴望钩，驱动点下一章"；以及承诺兑现概念(HARD-002 行 312 的定性核：上一章章末的承诺，下一章必须有实质回应，绝不能完全不兑现)。
【明确丢弃】(全是基础设施/解析层/CC 依赖，且撞 Loom 红线)：
- §4 全部约束分层(Hard Invariants / Soft Guidance / rationale_type 枚举 / 债务系统 / Override Contract)——这是打分/裁决引擎，Loom 红线③禁止打分引擎，且我们改"措辞用建议/自检问句不用硬门"。
- §5 兼容性说明(reviewer 维度表、hook_type/micro_payoffs 等输出字段 JSON 映射、checker 对接)——解析层，与 file-as-truth 的极简 skill 无关。
- §2.3 的 30/40/30 占比数字、各题材"强度建议 strong/medium"枚举——量化标准，只取"分工"定性，不搬数字门槛。
- 所有"Step 1.5 / Context Agent / webnovel-review Step 4"等 CC pipeline 引用。
注意红线①：微兑现/钩分工是 voice/手艺层(怎么写得让人追)，不是题材剧情套路(what)，所以进 skills(跨书复用手艺)是对的，不会污染设定师/世界观，也绝不进写作指纹。

## 一、loom/templates/skills/故事引擎.md 的三处改动

### 改动 A：在「## 情绪工程」小节末尾新增一条 bullet（接在 `- 压抑别超过本章——憋一段、爽一段。` 之后）

```markdown
- **微兑现(每章 1-2 个小收获)**:大爽点之外,本章再给读者 1-2 个轻量的"小收获",让人觉得"这章没白看"。可以是——揭一条新信息/线索、推进一段关系、涨一点能力或拿到一件东西、得一次认可长一次脸、回收一个早埋的伏笔、放一次憋住的情绪。它比大爽点轻、比大爽点勤,负责撑住章内的沉浸。**过渡章/铺垫章哪怕没有大爽点,也建议至少留 1 个微兑现**(给条新线索、添个小互动都行),别让读者空看一章。自检一句:读者看完这章,带走了什么?
```

### 改动 B：在「## 爽点类型」与「## 本章细纲要给到的」之间，新增独立小节

```markdown
## 钩子分工(章内 vs 章末)
钩子放在哪,职能不一样,别都堆到章末:
- **章内钩**(悬念/情绪):用在场景与场景之间,负责"留住读者别合上这一章"——抛个没解的疑问、点一下情绪,让人往下一段滑。
- **章末钩**(危机/抉择/期待·打脸前置):用在收尾,负责"驱动读者点开下一章"——它强度更高,是给下一章的一句承诺。
说得出来:这一章的章末钩,是哪一类?它向读者许了什么、要在下一章兑现?
```

### 改动 C：在「## 本章细纲要给到的」第 3 条「章首」尾部补半句

原文（:17）：
```markdown
3. **章首**:怎么快速接住上一章的钩子(别大段回顾)。
```
改为：
```markdown
3. **章首**:怎么快速接住上一章的钩子(别大段回顾)——要**真接住**(对上一章章末许的那个承诺给出实质回应),不是提一嘴就跳过。
```

## 二、loom/templates/skills/评估自检.md 的改动

在「## 逐项过(是非题,每条不过就改)」列表中，于现「钩子」条之后、「章首衔接」条之前，插入以下两条（与现有条目同构，用 `- [ ]` 是非题语气）：

```markdown
- [ ] **微兑现**:除了大爽点,这一章有没有给读者 1-2 个小收获(新信息/关系推进/涨能力/拿东西/被认可/回收伏笔/情绪释放)?一个都说不出来 = 读者这章白看了,补一个。过渡章至少留 1 个。
- [ ] **承诺兑现**:上一章章末的钩子,这一章实质接住/兑现了吗?只字未提或随手丢掉 = 失信于读者,补上回应(哪怕是"暂时按下、给个新进展"也行,别装作没埋过)。
```

修改后该列表完整顺序为：爽点 → 钩子 → 微兑现(新) → 承诺兑现(新) → 章首衔接 → 人物不 OOC → 设定不漂移 → 节奏 → 主线。
