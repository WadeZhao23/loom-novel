# 旅程编排不引入 LangGraph:手搓旅程状态机,照抄其协议形状、不引其依赖

> 状态:**已采纳**(设计见 [全程写作伙伴 spec](../superpowers/specs/2026-07-10-journey-partner-design.md),实现随一期落地)。

起因是「全程写作伙伴」需求:访谈式交互覆盖创作旅程(起书访谈 → 章前决策 → 写后引导),需要一层管理「作者走到哪、还欠哪些创作决策」的旅程状态。作者最初提议用 LangGraph(StateGraph + interrupt + SqliteSaver)做这层编排。经四个独立视角的分析(产品形态取证 / LangGraph 钢人辩护 / 工程经济实测 / 旅程图复杂度实测,2026-07-10),裁决为**不引入**——本文留档否决理由,防日后"要不要上框架"反复重议。

## 决定

旅程编排层手搓:`journey.py` 显式状态机(STAGES 冻结表,照抄 `agents.py` 的 STEPS/StepSpec 模式)+ 「领航员」agent(第六个角色)+ 薄游标(挂 `.loom_state.json`,可丢弃、从文件谓词重推)。**但把 LangGraph 验证过的四个协议思想照抄过来**:问题卡=纯数据 payload(不是代码分支)、答案经单点注入(`POST /api/journey/answer`)、游标持久化、「改答案」=改 md 文件即 fork(签名失配自动重跑)。

## 否决 LangGraph 的四组证据

**① 三条产品红线,checkpoint 全撞。** 产品形态(本地单机/零服务器/文件即真相/一人维护/HTTP 请求-响应短事务)推出三条红线:不得有第二份状态真相;必须容忍用户绕过旅程直接改文件(细纲 WYSIWYG 判例:「你改了它,引擎就按你的来」);不能依赖内存执行上下文活过两次问答之间(用户答完一题可能关机隔三天再来)。LangGraph 的持久化前提是 checkpointer——msgpack 二进制 blob,人不可读不可改;作者手改 `世界观.md`(产品明示欢迎)的瞬间 checkpoint 即馊,领航员会基于旧值提问。唯一防漂移办法是每个节点从 md 重新 hydrate——**写完 hydrator 那天,状态机已经手搓完了,checkpointer 沦为摆设**。混合方案(仅访谈层用框架)的框架收益恰好就是被双真相毒死的那一块。

**② interrupt() 无执行栈可挂。** 它解决「挂起**运行中的执行栈**等人回答」;Loom 的 HTTP 一问一答之间没有任何挂着的计算,要持久化的只是游标+已收答案(一个 50 行可落盘的 dict)。且 interrupt 节点 resume 时**从头重放**——预填选项的 LLM 调用若与 interrupt 同节点,每次答题重复计费,BYOK 用户直接看到账单;规避纪律(拆节点/durable task)一上,「一行原语」的卖点已不存在。

**③ 工程经济(实测,langgraph 1.2.9,2026-07-10 干净 venv 实装)。** 净新增 **18 个发行版 / 26MB**(现运行时直接依赖仅 7 个,树扩大约 60%);Loom.app 74MB 估涨至 ~95-100MB;**langsmith 遥测客户端被 langchain-core 无条件拉入**——对把零联网写进 [ADR 0003](0003-init-offline-network-split.md) 的产品是每次发版的审计面;`pydantic<3` 上限将在 pydantic 3 落地时把 fastapi 升级锁死;**claude/codex 两条 CLI 通道协议层面就没有 tool-call**(刻意压成纯文本补全,backends.py 反 agent 护栏),框架预制件(ToolNode/structured_output)对一半后端天然失效;历史节奏约每年 2-3 波破坏性变更 × 5 个 lang* 包 × 双平台 PyInstaller 重打包,永续税由一人扛。

**④ 旅程图复杂度远低于框架回本线。** 把全程伙伴的状态图真画出来:**12 个静态宏节点、~15 条边、0 并行、0 运行时动态路由、2 个 while 可表达的环**(章循环、gate 回炉——后者已用 11 行 for 实现)。段内 2-6 个问题是**数据**(领航员按 md 现状出题),不是拓扑节点。全图最吓人的转移「写到第 10 章回头改世界观」在文件真相架构里是 **0 条边**:改 md → 设定师 reads 签名失配 → 下次织章自动从第 0 棒全量重跑——**图框架需要显式建模的失效传播,被文件真相吸收成非事件**;引入框架不但省不掉这部分,反要为 checkpoint 第二真相重新发明它。库内先例:webui 五步旅程卡(app.js)的旅程状态全部是派生谓词,零存储指针。

## Considered Options(选摘)

- **A、LangGraph 全面接管**(旅程+流水线全迁 StateGraph)——否决:上述四组证据全中;另需重验证 agents.py/ledger/gates ~1500 行战功最多的代码,动最不该动的换不到用户可见收益。
- **B、手搓 journey.py(采纳)**——一次性 300-450 行(statebook.py 119 行/gates.py 160 行同粒度),零新依赖、零外部 churn,与 ledger/guard/原子写/事件契约全复用。
- **C、混合(流水线不动、仅访谈层用框架)**——否决:技术上干净可行(节点可直调 `backends.complete`,不碰 langchain 模型层),但吃下 A 的全部依赖税,而框架唯一收益(durable interrupt)被 md 双真相同步壳(~100+ 行)清零——**C = B 的代码量 + A 的依赖,两头付账**。钢人辩护的结案陈词:最小侵入方案收敛到极限就是 B。

## 红线

- **不引 langgraph/langchain 任何包**;若他日出现真实的高分支/并行编排需求,先怀疑违反了「交互从简」、该砍需求,再议框架。
- **旅程状态无第二真相**:游标之外一切派生自文件;游标可丢弃(坏了当无,谓词重推,同 ledger「坏 JSON 当无账本」哲学)。
- **协议形状常青**:问题卡=数据、答案单点注入、游标持久化、改文件=fork——这四条是从 LangGraph 吊来的书,换实现也不许丢。
