# 立项卡:平台/分区/对标定位从"零概念"落成人手维护的外置大脑一卡

> 状态:**已采纳并实现于 3.0.0**(立项卡模板 + 设定师 reads + sensitive 平台二档联动 + doctor 软检查 + server 展示)。

起因是重新起书时,作者的真实流程是"先定平台 → 分区 → 题材 → 找对标",而 Loom 原本对**平台/分区零表示**:`config.py` 的 `Config` 无字段、`scaffold.init()` 只吃 `name/parent/genre`、CLI 与 web 都没有 prompt;平台名(起点/番茄/晋江/七猫)在整个代码库里只作**两处文档字符串**出现——`违禁词.md` 的"审核松紧"警告、`archive.py` 的导出目的地备注,没有任何逻辑按它分支。定位是**不能跳的作者决策**,但此前无处落。

## 决定:加一份人手维护的「立项卡」,不联网、不建映射树

外置大脑新增一份**可选**的立项卡,人手填 `平台 / 分区 / 题材 / 对标意图 / 为什么选它`,各格可空、留空照常出稿。**只**两个消费者,别的一律不碰:

- **平台** → 给 `违禁词.md` 定基线松紧(起点/番茄严、晋江另一套),是全库**唯一**对平台敏感的联动面;
- **整卡** → 喂设定师当定位背景(管 what/where);**绝不进写手/写作指纹**,不污染 voice。

分区改为**可选、用户自填的自由文本**;对标素材由**用户自己找、自己贴**(喂给 `拆书`),立项卡本身只记"选了谁、为什么"。

## Considered Options(选摘)

- **`loom.toml` 加 `平台/分区` 字段**——否决:流水线没有任何一棒会读,是死数据;且会诱使日后写"按平台分支"的逻辑,违 Loom 全程"粗粒度一刀切、不按平台特判"。定位是给人和设定师看的散文,不是配置。
- **`平台→分区→题材` 级联映射树(静态白名单)**——否决:平台分区是会过时的白名单,跟模型路由那条"别写死会过时白名单"同源;还逼 Loom 维护一套逐平台分类学。
- **起书时联网实时拉分区(照搬模型路由的"拉取可用模型")**——否决:模型路由能 live-fetch 靠三个前提——① 标准稳定 API(`GET /models`)② 数据真 churn(模型天天出新)③ 在设置页手动触发;平台分区**三条全不满足**:无公开分区 API 只能脆爬 HTML(=逐平台分支逻辑)、分区几年才动一次(staleness≈0)、且要在 `loom init` 里拉。而 `loom init` 的**零联网离线**是刻意不变量(断网也能起书),不为一个几年不变的清单去砸它。分区永远最新靠**人手填**——作者本就在那个平台发文,比任何爬虫都清楚,同"素材靠用户提供"哲学(见 [ADR 0003](0003-init-offline-network-split.md))。
- **Loom 帮找/收集/排序对标作品**——否决:Loom 的 `拆书` 只吃你本地贴的**单份**文本、抽可迁移条件框架、强制专名黑名单、绝不碰 voice、输出落隔离 `.拆书/`(无 agent 读、需人手挑进世界观)。收集/发现/排序归人,立项卡只记"选了谁、为什么"。

## 红线

- **不回写**:loom 从不自动写立项卡(同 `违禁词.md`——外置大脑里人手维护、只读不回写的那类);它不随故事生长、没有 `[AI补充]`。
- **不越 voice 线**:整卡只喂设定师(what/where),绝不进写手/写作指纹(守 [ADR 0002](0002-fingerprint-purpose-and-division.md) 的 voice/what 分离)。
- **不阻断**:缺卡/空卡一律照常出稿;平台联动只调违禁词松紧、非阻断(守 [ADR 0006](0006-quality-gates-issue-driven-non-blocking.md))。
