# 细纲落盘成可看可改的文件,大纲师对它 WYSIWYG:存在就读它,不再重生成

把大纲师产出的**分镜细纲**(本章场景骨头)从"用完即弃的工作区中间态"升级成**第一类可编辑产物**:落盘到 `正文/.细纲/第N章.md`,作者可看可改。规则只有一句——**这文件存在,大纲师就读它**(WYSIWYG),不再调 LLM 重新生成;清空它或点「重新生成细纲」才会重生。

回应作者诉求:"每一章的细纲能看/能改吗?"——能。细纲是这本书结构的控制面,改它即改这一章的骨架。

## 之前 vs 现在

- **之前**:大纲师每跑一次都现编一份细纲,只进**本章工作区**(给写手/编辑/润色师读)+ `ledger.json`(纯为断点续跑),跑完即弃。作者**看不到、改不了**;想调结构只能改卡章纲那一句话,粒度太粗。
- **现在**:大纲师首次生成细纲时**额外落盘** `正文/.细纲/第N章.md`。此后任何重跑都**优先读这份文件**:
  - 你**手改了**它 → 重写本章按你的细纲走(写手读到的就是你改的)。
  - 你**没动**它 → 重写也沿用它(结构稳定,只重写文笔)。
  - 想要**新方案** → 「重新生成细纲」跑 设定师→大纲师 覆盖它(此刻读到最新的卡章纲/世界观/人物卡);或在编辑器里清空它,下次写章即重生。

## 为什么是 WYSIWYG(存在即读),而不是"没手改就重生成"

考虑过用快照对比"作者有没有手改细纲、只在改过时才沿用"。否决:① 要再引一份 `.细纲快照` + 漂移判定,复杂度向章节 `.原稿` 看齐却没那么必要;② 语义绕——作者看到编辑器里的细纲,却可能在重写时被悄悄换掉,违直觉。**WYSIWYG 最简单可预期:你看到的细纲,就是重写本章会用的细纲。** 想换结构的入口很明确(「重新生成细纲」按钮 / 清空文件),不靠"猜你改没改"。

代价:改了上游(卡章纲/世界观)却没动细纲时,重写本章不会自动反映——因为细纲把它们"定格"了。这是 WYSIWYG 的必然取舍,用「重新生成细纲」一键解决(它会重读上游)。已在 UI 文案和按钮 tooltip 里讲明。

## 与既有不变量的兼容

- **ledger/续跑**:大纲师无论"读文件"还是"生成",产出照样 `record_step` 进 ledger;下游写手的上游签名含大纲师产物,改了细纲→写手自动重算。不新增工序,PIPELINE 仍是 5 棒。
- **章节管理**:`正文/.细纲/第N章.md` 已加入 `chapters._ARTIFACTS`,删 / 插空章 / 上下移都随整章产物一起**安全重编号 + 进回收站**,不丢、不串号。
- **后端协议不变**:「重新生成细纲」(`/api/outline/regen`)只用 `backend.complete`,跑 设定师→大纲师 两棒;DeepSeek/claude/codex 通吃。
- **正交于 voice/what**:细纲只管这一章的结构骨架,绝不回流写作指纹;与卡章纲(跨章脊柱 + `[AI回顾]`)分工清楚,不混。

## Considered Options

- **细纲只读展示、不可改**——否决:满足不了"能改"。
- **细纲可改、但每次重写都重新生成(只读展示最近一版)**——否决:作者的手改会被下次重写冲掉,等于没法改。
- **快照对比、仅在手改时沿用**——否决:复杂且反直觉(见上)。
- **WYSIWYG:落盘可改,存在即读,显式重生成**(选中)——最简单可预期,作者对结构有真正掌控,且与 ledger/章节管理/后端协议全兼容。
