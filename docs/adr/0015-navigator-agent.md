# 书房伙伴:对话 agent 吞访谈,领航员从出题机升级为工具循环

> 状态:**已采纳**(设计见 [书房伙伴设计 spec](../superpowers/specs/2026-07-16-navigator-agent-design.md);继承 [两层卡设计](../superpowers/specs/2026-07-15-navigator-two-layer-card-design.md) 的两件设计遗产——槽位扫描器与三期定址落盘表)。

起因是作者拍板(2026-07-16):领航员从「出题卡片机」(ADR 0013 手搓的旅程状态机 + 问题卡)升级为**对话式 agent**——访谈本身也走对话,问题卡片机退役;agent 手长出工具(读文件/看地基/提设定,分期到开工序/查资料);对话记录存书内 `.伙伴对话/`。本 ADR 记录这次裁决与随之而来的协议形状换代,留档防日后「要不要用对话循环替代问答卡」反复重议。

## 决定

领航员升级为架在五后端共用单轮 `complete()` 之上的**手搓对话循环**:每轮 = 一次 HTTP 往返(`POST /api/partner/say`,ndjson 流)——读 `.伙伴对话/*.jsonl` 尾部 + 书文件现状(谓词/槽位现算)→ 拼 prompt → `backend.complete()` → 解析说话段 + 至多一个文本协议工具块(`用:<工具名>`)→ 有工具则执行并回喂(≤6 次/轮)、无工具则本轮终结。

**核心不变量原样继承 ADR 0013**:两轮之间进程里没有任何活着的计算——关机三天回来,下一轮从 jsonl 尾部 + 文件现状重建一切;人在回路(拍板)发生在两轮之间,不是挂起的 interrupt。本设计换的只是协议的**形状**(问答卡 → 说话+工具),不是这条不变量本身。

## 继承 ADR 0013,协议形状换代

ADR 0013 否决 LangGraph 的三条产品红线——不得有第二份状态真相、必须容忍用户绕过旅程直接改文件、不能依赖内存执行上下文活过两次问答之间——与「手搓不引框架」的决定**原样继承**。ADR 0013 的访谈编排部分(问题卡=数据 payload、`journey.goto` 聚焦流、游标存跳过标记与各段已问计数)随本 ADR **标注被取代**,四条协议形状按同一条纪律换代:

- **消息=数据**:取代「问题卡=纯数据 payload」。对话事件(`user`/`assistant`/`tool`/`result`/`proposal`/`confirm`/`meta`/`summary`)是 `.伙伴对话/*.jsonl` 里的纯数据行,不是代码分支;解析/执行/渲染三处共用同一份工具注册表(单一真相),prompt 与 parser 永不漂移。
- **拍板单点落盘**:取代「答案单点注入(`POST /api/journey/answer`)」。`POST /api/partner/confirm {root, id}` 是唯一落笔入口——agent 发 `提设定` 只产出候选卡(proposal 事件),不写盘;proposal 未被作者点一律不落盘,幂等(重复确认直接返回已落盘结果、绝不二次写)。
- **对话可丢弃**:取代「游标持久化(`.loom_state.json`)」。`.伙伴对话/` 整个删掉,书完好无损——未拍板的候选随之作废,伙伴失忆,仅此而已。完成度/门禁/游标永远从书文件现状派生,上下文≠状态;唯一被程序读回的对话内容是**待确认 proposal 行**(ADR 0013 薄游标「待答卡缓存」的 jsonl 同构:可丢弃、不派生完成度)。
- **改文件即分叉**:取代「改答案=改 md 文件即 fork(签名失配自动重跑)」。confirm 时落点现状与候选卡快照不符(作者拍板前手改了那一格)→ 拒绝落盘并告知,伙伴下一轮看见结果重新提——文件真相赢,永远。

## 与 ADR 0014 / 0011 的关系

- ADR 0014 立项红线第三条「诊断/AI 不碰立项卡:立项永远走访谈/手填/建书代落」的「访谈」通道由本 ADR 的对话 agent 承接;原辩护措辞「不是 AI 发明」由本 ADR 的「**候选卡结构性可见 + 拍板才落盘**」取代——判据从「是不是 AI 想的」挪到「点没点确认」,AI 起草的候选未经拍板一个字不进立项卡。ADR 0014 的红线条文同步修订(见该文件)。
- ADR 0011「不回写:loom 从不自动写立项卡」保留;拍板代落**不算**自动写(判据=没点就不落盘,同「建书代落平台/题材」先例)。ADR 0011「收集/发现/排序归人」条款留给 v1c 查资料工具单独修订注(检索可由伙伴代劳、带来源,采信与排序仍归人)。

## 为什么不建向量库(裁决摘要,重议触发器见下)

外置大脑是几十个百来 KB 的 md,文件名即语义索引(`主角·林潜.md`/`金手指.md`)、槽位扫描器给出容器×键的精确结构——向量检索是「无结构时对结构的近似模拟」,这里结构真实存在,用近似替精确是倒退。且嵌入索引是文件内容的派生缓存,作者改一份 md 即馊,是全系统里唯一没法用 sig 哈希 + 谓词重推兜底的第二真相,与 langgraph checkpoint 同病;五后端里 Claude/Codex CLI 无 embedding 端点、demo 无网,本地嵌入模型又给 25MB 包塞几百 MB。正文长大的跨章记忆已有确定性解(卡章纲脊柱 + `[AI回顾]` + 状态账本滚动窗,人可读可改的文件,非黑盒索引)。

**重议触发器**:当且仅当 (a) 给注册表加的 `搜正文`(grep)工具在真机上明显不够(作者的模糊回忆 grep 命不中),**且** (b) 正文规模已过数百章——届时嵌入索引作为**可丢弃的派生缓存**(删了 grep 重建、绝不当真相、绝不进 sig/门禁)再议。在此之前,加向量库 = 违本 ADR 与 ADR 0013 的无第二真相红线。

## Considered Options(选摘)

- **继续用/改引 LangGraph 或其他编排框架承接对话循环**——否决:ADR 0013 的四组证据(三条产品红线撞 checkpoint / interrupt 无执行栈可挂 / 工程经济 / 图复杂度远低于回本线)原样成立,本设计只是把编排对象从「问答卡」换成「说话+工具」,框架收益的结构没变。
- **给对话 agent 建向量库检索外置大脑**——否决:见上「为什么不建向量库」。
- **CLI 后端沿用现有反 agent 护栏、不做伙伴变体**——否决:`ClaudeCodeBackend`/`CodexBackend` 现有护栏每次调用注入「禁止调用或提及任何工具、禁止反问」,会掐死文本协议工具块与「向作者发问」这个本职。改为 `agent_mode` 开关的伙伴专用变体:允许输出至多一个 `用:` 块、允许反问;真实工具执行仍留在 loom 服务端,`--allowed-tools ""` 与 codex 只读沙箱不松。
- **预填多候选一键拍板(沿用问题卡的 2-4 选项形态)**——暂缓,列入诚实取舍:候选卡 v1a 只承接单提案,预填多选项退化为打字作答或单候选确认;`_MAX_QUESTIONS=4` 式硬收敛保证随之消失,访谈长度改由 persona 调教(留痕收真机轮数数据,若实测收敛差再议多提案候选卡)。

## 红线

- **不引 langgraph/langchain 等编排框架**;协议手搓,原样继承 ADR 0013「不引框架」的决定。
- **无第二状态真相**:`.伙伴对话/` 可丢弃,完成度/门禁/游标永远从书文件现状派生;唯一被程序读回的对话内容是待确认 proposal 行(过期归档失效/重复确认幂等/落点快照失配拒绝三条判据)。
- **候选卡结构性可见**:proposal 必须带落点现状(旧值,已填时确认键从「就这么定」改「替换」)、卡面虚线边 + 「候选」角标,作者拍板前不落盘——这是唯一挡住 AI 文字混进书的结构性保证,不靠 agent 自觉。
- **两轮之间零挂起**:每轮从 jsonl 尾部 + 文件现状重建,不依赖任何进程内存活的执行栈或计时器;崩溃语义平庸化,任何时刻杀进程最多丢正在写的一行。
- **不建向量库**:见上,重议触发器未满足前维持决定。
